一、核心问题：死循环 vs. Agent Harness

🎯 答案：不是死循环，是带 Harness 的 Agent 编排

Ironsmith 的生成管道不是简单的 "while(有错误) { 修复 }" 死循环，而是一个 由多个专用 Agent 组成的编排系统，每个 Agent 有明确的职责边界、输入/输出契约和故障恢复策略。整体由 SingleFileToolGenerationRuntime 这个 orchestrator 调度，但真正的复杂度在 ContentViewBuildRepairLoop 这个 Agent 内部。

Agent 编排 Harness 控制 策略选择 故障恢复

具体来说，Ironsmith 的架构是 "Controller + Specialized Agents + Deterministic Harness" 三层结构：

🎛️ Controller（调度层）

SingleFileToolGenerationRuntime — 负责六阶段顺序执行、状态机管理、失败重试和回退策略。它知道每个阶段该调用哪个 Agent，但不懂具体怎么生成代码。

相当于 "项目经理"，知道流程，不懂技术细节

🤖 Specialized Agents（执行层）

ToolMetadataClient（元数据 Agent）、ContentViewBuildRepairLoop（编译修复 Agent）、ToolAppBundleClient（打包 Agent）等。每个 Agent 有独立的提示词、工具使用和记忆策略。

相当于 "各专业工程师"，各司其职

🔧 Deterministic Harness（可靠性层）

ContentViewSourceCleanup（正则清洗）、ToolGenerationRepairPolicy（策略配置）、ToolVersionBackupClient（版本备份）。这些不是 LLM，是代码逻辑，确保 Agent 的输出不会破坏系统。

相当于 "质检员 + 安全员"，用代码规则保护系统

关键洞察： 如果只用死循环，系统会在错误修复时无限震荡。Ironsmith 通过引入 ToolRepairStrategy（策略选择）和 ToolGenerationRepairPolicy（策略配置），把 "修复" 变成一个有策略、有预算、有回退的 Agent 行为。

二、六阶段生成管道全景图

User Prompt ↓ [1] Metadata Generation (ToolMetadataClient) ↓ [2] Prompt Refinement (可选，由 LLM 扩展) ↓ [3] Content Generation (LLM 写 ContentView.swift) ↓ [4] Source Cleanup (ContentViewSourceCleanup — 正则清洗) ↓ [5] Compilation + Repair Loop (ContentViewBuildRepairLoop — 核心 Agent) ↓ [6] App Bundle Packaging (ToolAppBundleClient — 签名、沙盒、图标) ↓ 🎉 App Bundle (可直接运行)

每个阶段都有明确的 输入契约 和 输出契约，如果输出不符合契约，Controller 会触发回退策略。

阶段 1：Metadata Generation（元数据生成）

Agent： ToolMetadataClient

输入： 用户原始 Prompt

输出： 短显示名称 + 图标描述

特点： 可以用轻量级本地模型（如 Apple Foundation Model）做结构化生成，有确定性回退。如果元数据生成失败，不影响后续流程。

💡 为什么单独分一个阶段？ 成本优化。元数据不需要 GPT-4 级别的模型，本地小模型就能做，省 API 费用。如果本地模型失败，用确定性回退（如从用户 prompt 中提取关键词）。

阶段 2：Prompt Refinement（提示词优化）

Agent： 可选的 LLM 阶段

输入： 用户原始 Prompt

输出： 扩展后的 Prompt（添加实现细节）

控制： 由 promptRefinementEnabled 标志控制。如果关闭，直接用用户原始 Prompt 进入阶段 3。

💡 这是一个 "可选 Agent"。如果用户描述很清晰，可以跳过；如果用户描述模糊，LLM 会补充细节（如 "添加搜索和筛选功能"）。

阶段 3：Content Generation（核心代码生成）

Agent： 选中的 LLM（DeepSeek / GPT-4o / Claude 等）

输入： 系统提示（约 40 条约束）+ 优化后的用户 Prompt

输出： 完整的 ContentView.swift 代码

关键约束：

  • "Write exactly one complete Swift file named ContentView.swift"
  • "Define exactly one View-conforming type: struct ContentView: View"
  • "Do not use ObservableObject, @Published, @StateObject, or @ObservedObject"
  • "Keep state directly inside ContentView with @State properties"
  • "Don't make anything overly complex. The code needs to fit in a single file"
  • "Prefer Apple platform frameworks and native APIs when they fit"

这些约束不是建议，是 硬性规则。如果 LLM 违反，阶段 4 的 cleanup 会强制修正。

阶段 4：Source Cleanup（源代码清洗）

Harness： ContentViewSourceCleanup（纯代码，不是 LLM）

功能：

  • 剥离 Markdown 围栏、think 块、脚手架
  • 移除 LLM 生成的 App/Preview 块（即使系统提示禁止，LLM 有时仍会添加）
  • 规范化 import 语句，修复 macOS SwiftUI 常见陷阱
  • 将松散顶层状态移入 ContentView
  • 移除错位成员作用域视图块
  • 运行 swift-format 统一格式

🔧 这是 Deterministic Harness 的关键。LLM 输出不可靠，但代码规则的 cleanup 是 100% 可靠的。这是 Ironsmith 稳定性的第一道防线。

阶段 5：Compilation + Repair Loop（编译修复循环）

Agent： ContentViewBuildRepairLoop（最复杂的 Agent）

详见 第七节。

阶段 6：App Bundle Packaging（应用打包）

Agent： ToolAppBundleClient

功能： 构建内部应用包（LSUIElement，无 Dock 图标）或可导出到 /Applications 的完整包。包括签名、沙盒权限、图标生成。

三、提示词编排：系统约束与分层策略

3.1 系统提示的架构：约 40 条约束

Ironsmith 的系统提示不是一段自然语言描述，而是一个 高度结构化的约束列表，定义了 LLM 可以做什么和不能做什么。这是架构的边界。

系统提示核心结构（推测，基于 AGENTS.md 约束反推）
You are a SwiftUI developer. Write a single file named ContentView.swift.

[框架规则]
- Define exactly one struct: ContentView: View
- Do NOT create @main, App, Preview, or #Preview blocks
- Do NOT use ObservableObject, @Published, @StateObject, @ObservedObject
- Keep all state with @State inside ContentView
- Platform: macOS 26.0+ only

[代码组织]
Use these sections:
// MARK: - State
// MARK: - Body
// MARK: - Actions
// MARK: - Helpers

[功能约束]
- No backend, no analytics, no iCloud
- No networking unless explicitly requested
- Use native Apple frameworks when possible
- Keep it simple enough for a single file

[输出格式]
Return ONLY Swift source code. No markdown fences.
No explanation text. No thinking blocks.

3.2 分层提示策略：不同 Agent 用不同提示

Ironsmith 不是用一个 "万能提示" 解决所有问题，而是 每个 Agent 有自己的提示词。

📝 ToolMetadataClient（元数据 Agent）

提示词：轻量级，结构化输出（JSON 或 Swift struct）。要求生成短名称和图标描述。

可用本地模型（Apple Foundation Model）或弱 LLM

📝 代码生成 Agent（阶段 3）

提示词：约 40 条约束，包含框架规则、代码组织、功能约束、输出格式。是最长的系统提示。

需要强 LLM（GPT-4o / Claude / DeepSeek）

📝 修复 Agent（阶段 5）

提示词：包含编译器错误诊断、当前源代码、之前修复的结果（上下文）。要求返回 unified diff 或完整重写。

根据模型能力选择 diff 或 full-rewrite 策略

📝 编辑 Agent（Edit Mode）

提示词：包含当前代码 + 用户修改指令。如果模型支持 diff，要求 bounded unified diff；如果不支持，要求完整重写。

diff 策略：减少 token 消耗，保留未修改部分

3.3 提示词中的 "负约束" 设计

Ironsmith 的提示词特别注重 "不要做什么"，而不是只告诉 LLM "做什么"。这是因为 LLM 有强烈的 "过度工程化" 倾向。

关键负约束：

"Do not create preview providers or #Preview blocks" — LLM 经常忍不住加 preview，但 preview 会干扰编译。

"Do not append @main to any struct" — LLM 有时会尝试写完整的 App 结构，但 @main 入口由平台提供。

"No backend, no analytics, no iCloud" — 明确限制 LLM 的 "想象力"，防止生成过度复杂的架构。

四、Tool 使用：谁有工具调用能力

Ironsmith 的架构中，Tool 的使用方式非常明确：不是让 LLM 调用外部工具，而是让 Harness 代码调用 LLM，然后处理输出。这是 "LLM 作为工具" 而不是 "LLM 使用工具" 的架构。

4.1 没有 Tool Calling（Function Calling）

在 Ironsmith 的生成管道中，LLM 本身不调用外部工具。LLM 的唯一输出是文本（代码或 diff）。所有工具调用（编译、文件操作、格式化、签名）由 Harness 代码完成。

🔧 SwiftPackageProcessClient（编译工具）

由 Harness 代码调用，执行 swift build。不是 LLM 调用。

输入：源代码文件 | 输出：编译器诊断信息

🔧 swift-format（格式化工具）

由 ContentViewSourceCleanup 调用。纯代码调用。

输入：原始代码 | 输出：格式化后的代码

🔧 codesign（签名工具）

由 ToolAppBundleClient 调用。纯代码调用。

输入：app bundle | 输出：签名后的 bundle

🔧 xattr（隔离属性清除）

由 ToolAppBundleClient 调用。清除 macOS 的 quarantine 属性。

纯系统调用，与 LLM 无关

4.2 LLM 作为 "被调用的工具"

在 Ironsmith 的架构中，LLM 是 被 Harness 调用的一个能力，而不是主动调用其他工具的 Agent。

Harness 代码（Controller） ↓ 调用 LLM API（生成代码） ↓ 返回文本 Harness 代码（处理输出） ↓ 调用 编译工具 / 文件系统 / 格式化工具 ↓ 结果 Harness 代码（判断下一步）

🔑 关键设计原则： LLM 只负责 "生成文本"，Harness 负责 "执行动作"。这种分离避免了 LLM 的幻觉导致危险操作（如删除文件、执行恶意代码）。

4.3 唯一的 "Tool" 是 LLM 本身

如果非要定义 Tool，那么 Ironsmith 中唯一的 Tool 是 LanguageModelClient，它封装了所有 LLM 调用：

  • LanguageModelClient — 构造 AnyLanguageModel 会话
  • RemoteModelClient — 获取远程模型列表
  • OllamaClient — 检测/启动本地 Ollama
  • LocalModelClient — 下载 MLX 模型

但这些 "Tool" 是 供人类用户配置和选择模型 用的，不是 LLM 在生成过程中调用的。

五、Memory 机制：上下文与记忆管理

5.1 没有持久化 Memory（数据库/向量存储）

Ironsmith 的生成管道 不使用外部记忆系统（如向量数据库、长期记忆）。所有 "记忆" 都是 即时上下文（in-context）的，通过对话历史传递。

5.2 三种上下文传递机制

📜 对话历史（Conversation History）

在修复循环中，之前的编译错误和修复结果会被包含在下一次请求的上下文中。这让 LLM 能 "学习" 之前的错误模式。

用于：编译修复循环 | 限制：受上下文窗口限制

📜 版本暂存（Staged Backup）

编辑模式下，当前代码被暂存到 .ironsmith/versions/pending-ContentView.swift。这是 "短期记忆"，用于编辑失败时恢复。

用于：编辑模式 | 生命周期：一次编辑操作

📜 上一版本（Previous Version）

编辑成功后，旧代码被保存为 previous-ContentView.swift。这是 "长期记忆"，允许用户一键恢复。

用于：版本恢复 | 生命周期：直到下一次编辑成功

5.3 上下文窗口管理

修复循环中，如果对话历史超过上下文窗口，系统会 压缩一次（compact）。如果仍然超过，则触发 重新生成（从头开始，丢弃所有修复历史）。

💡 这是一个务实的策略。不追求完美的长期记忆，而是保证 "在当前任务内能完成"。如果记忆太长了，放弃记忆，重新开始。

5.4 没有跨任务记忆

Ironsmith 不会记录 "用户之前生成过什么类型的应用" 或 "用户偏好什么风格"。每个生成任务都是独立的。如果需要个性化，用户需要在 Prompt 中明确表达。

六、Harness 详解：每个阶段的控制器

Harness 是 Ironsmith 架构中最被低估的部分。它不是 "辅助工具"，而是 确保系统可靠性的核心机制。

6.1 ContentViewSourceCleanup（清洗 Harness）

类型： 纯代码规则（正则 + AST 操作）

职责：

  • 剥离 Markdown 围栏（```swift 等）
  • 移除 think 块（<think> 或 reasoning 内容）
  • 移除 LLM 生成的 App/Preview 结构（即使被禁止，LLM 仍会偶尔添加）
  • 规范化 import 语句顺序
  • 修复常见 macOS SwiftUI 陷阱（如错误的修饰符用法）
  • 将松散顶层变量移入 ContentView
  • 运行 swift-format 统一格式

为什么重要： LLM 输出不可靠。即使系统提示说 "不要加 Preview"，LLM 仍有 5-10% 的概率会加。Cleanup 是 100% 可靠的修正层。

6.2 ToolGenerationRepairPolicy（策略配置 Harness）

类型： 配置对象

职责： 定义所有数字阈值和策略预算：

  • 最大生成尝试次数（如 3 次）
  • 确定性修复最大轮数（如 5 轮）
  • 初始编辑 diff 的 hunk 限制
  • 修复 diff 的 hunk 限制
  • 无效 patch 的最大容忍次数（stall 阈值）
  • 模型修复预算（token 或轮数）

为什么重要： 防止无限循环。如果没有任何策略限制，修复 Agent 可能在错误震荡中消耗无限 API token。

6.3 ToolVersionBackupClient（版本备份 Harness）

类型： 文件系统操作

职责：

  • 编辑前：将当前代码复制到 pending-ContentView.swift
  • 编辑成功：将 pending 提升为 previous（旧版本成为备份）
  • 编辑失败：从 pending 恢复原始代码，删除暂存
  • 路径逃逸检查：确保所有文件操作在包目录内

为什么重要： 原子性。编辑是一个可能破坏代码的操作，版本备份确保 "要么成功，要么恢复原状"。

6.4 SwiftPackageProcessClient（编译 Harness）

类型： 进程调用

职责： 执行 swift build，捕获 stdout/stderr，解析编译器诊断。

输出： 结构化的 SwiftCompilerDiagnostic 数组，只包含与 ContentView.swift 相关的错误。

6.5 AgentDiagnosticsLog（诊断日志 Harness）

类型： 日志记录

职责： 记录修复循环的每一步（DEBUG 模式，仅本地文件 ~/.ironsmith/agent-diagnostics.log）。

限制： 日志紧凑，不存储完整模型响应。仅用于解释修复循环行为。

七、编译-修复循环：最复杂的 Agent

这是 Ironsmith 的核心，也是最能回答 "是死循环还是 Agent" 的部分。

7.1 流程：不是死循环，是状态机

生成候选代码 ↓ 清洗 + 格式化 ↓ 编译（swift build） ↓ 解析诊断 ↓ 确定性修复 (正则规则，最多 N 轮) ↓ 如果仍有错误 模型修复 (LLM 生成 diff，验证后应用) ↓ 如果仍然失败 重新生成 (从头开始，最多 M 次) ↓ 如果全部失败 回退最佳候选 (返回错误最少的那次结果)

7.2 确定性修复（Deterministic Repair）

不是 LLM，是代码规则。 针对常见 Swift/SwiftUI 错误，用正则表达式或 AST 操作自动修复。

示例修复规则：

  • 错误的修饰符语法（如 .padding(16) 应为 .padding(16)）
  • 缺失的 import（如使用了 WebKit 但没有 import WebKit）
  • 常见的 API 不匹配（如错误的初始化器签名）

关键设计： 修复规则必须是 "广泛适用的 Swift 修正"，而不是 "特定应用的实现"。如果修复规则是针对某个特定应用的，它会破坏其他应用。

🛡️ 安全原则： 每个修复规则都有对应的测试。如果确定性编辑失败，系统是安全的（可以跳过或回滚）。

7.3 模型修复（Model Repair）

使用 LLM，但严格控制。

输入：

  • 编译器诊断（错误信息 + 行号）
  • 当前源代码
  • 之前修复的结果（用于避免重复同样的错误）

输出格式要求：

  • Unified Diff（如果模型支持）：只改有问题的部分，保留其他代码
  • 完整重写（如果模型不支持 diff）：重写整个 ContentView.swift

验证机制：

  • 应用 diff 后重新编译
  • 如果错误数增加 → 回滚（rollback）
  • 如果编译结果变成了占位符脚手架 → 回滚
  • 如果 patch 无效 → 计数 +1，达到 stall 阈值后放弃模型修复

7.4 重新生成（Regeneration Fallback）

当以下情况发生时，系统放弃修复，从头重新生成：

  • 错误数超过阈值（代码太烂，修不好）
  • 确定性修复陷入僵局（循环修复同一错误）
  • 模型 patch 反复无效（LLM 无法理解错误）
  • 上下文窗口超限（记忆太长，压缩后仍然超限）

策略： 重新生成时保留之前最好的候选（错误最少的那次），如果最终全部失败，返回这个最佳候选而不是彻底失败。

🎯 核心设计：渐进式回退。 不是 "成功或失败" 的二元结果，而是 "尽力做到最好，然后返回最好的结果"。

八、编辑模式：Diff vs. 全量重写

8.1 两种编辑策略

Ironsmith 的编辑模式不是简单地 "把 Prompt 发给 LLM 让它重写"，而是 根据模型能力选择最优策略。

📝 Diff 策略（Bounded Unified Diff）

适用模型： 支持长上下文和精确编辑的模型（如 GPT-4o、Claude 3.5 Sonnet）

流程：

  1. 暂存当前代码到 pending-ContentView.swift
  2. 发送 "当前代码 + 修改指令" 给 LLM
  3. 要求 LLM 返回 unified diff（有明确的修改范围）
  4. 验证 diff：应用后编译，检查错误
  5. 成功：promote pending 到 previous
  6. 失败：从 pending 恢复原始代码
优点：节省 token（只改需要改的部分）

📝 全量重写策略（Full Rewrite）

适用模型： 不支持 diff 或上下文较短的模型（如本地小模型）

流程：

  1. 暂存当前代码到 pending-ContentView.swift
  2. 发送 "当前代码 + 修改指令" 给 LLM
  3. 要求 LLM 返回完整的 ContentView.swift
  4. 验证：编译，检查错误
  5. 成功/失败：同上
缺点：消耗更多 token，但更简单可靠

8.2 策略选择：基于模型能力，不是模型名称

这是 Ironsmith 架构的关键设计：不根据模型名称选择策略，而是根据模型能力。

ToolRepairStrategy 由 InferenceStore 从 模型来源和声明的能力 推导：

  • 模型支持 diff 修复？→ Diff 策略
  • 模型只支持全量重写？→ Full Rewrite 策略
  • 模型的上下文窗口多大？→ 决定修复预算

💡 这意味着添加新模型（如 DeepSeek、通义千问）时，不需要改代码，只需要在 ProviderCatalog 中声明能力即可。

九、与 LongMarch 的对比启示

9.1 架构层面的差异

🎯 单文件约束

Ironsmith：LLM 只生成 ContentView.swift，框架由平台提供。
LongMarch：目前 LLM 生成完整 HTML，但应该约束为只生成 body 内容。

建议：提供固定 HTML 模板，LLM 只填充 body

🎯 编译-修复循环

Ironsmith：Swift 编译器 + 确定性修复 + 模型修复。
LongMarch：没有验证层，LLM 输出直接保存。

建议：添加 HTML/CSS/JS 语法验证 + 修复循环

🎯 确定性 Harness

Ironsmith：ContentViewSourceCleanup 用正则清洗 LLM 输出。
LongMarch：没有 cleanup 层，LLM 输出直接存储。

建议：添加 HTML 清洗层（移除 think 块、规范化标签等）

🎯 版本历史

Ironsmith：简单的前一版本备份（previous-ContentView.swift）。
LongMarch：没有版本历史。

建议：编辑前保存当前版本，支持一键恢复

🎯 多模型策略

Ironsmith：根据模型能力选择修复策略（diff vs 重写）。
LongMarch：只支持单一模型（DeepSeek）。

建议：支持用户自选模型（BYOK），根据能力选择策略

9.2 提示词层面的差异

📝 负约束

Ironsmith：约 40 条约束，其中一半以上是 "不要做什么"。
LongMarch：提示词较简短，缺乏负约束。

建议：添加 "不要包含 <html>/<head>" 等负约束

📝 分层提示

Ironsmith：不同 Agent 有不同提示（元数据、代码、修复、编辑）。
LongMarch：只有一个生成提示。

建议：分离元数据生成和代码生成的提示

9.3 最值得立即借鉴的 3 个点

  1. HTML 验证 + 修复循环： 在保存到数据库之前，用 HTML 解析器验证代码，修复常见错误（未闭合标签、错误嵌套等）。
  2. 源代码清洗层： 在 LLM 输出和保存之间，添加一个 cleanup 层，移除 think 块、Markdown 围栏、多余的 HTML 标签。
  3. 编辑前自动备份： 用户编辑应用时，自动保存当前版本。如果编辑失败或用户不满意，可以一键恢复。
