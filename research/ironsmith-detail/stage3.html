Agent（LLM 驱动）· 核心 · 最长提示词

职责

生成应用的 唯一可编辑文件：ContentView.swift。在约 40 条约束下，LLM 输出完整的 SwiftUI 代码。这是整个管道中最关键的 Agent。

源码结构

// SingleFileToolGenerationRuntime.swift
// 职责：六阶段管道 orchestrator，协调元数据→生成→清洗→修复→打包

struct SingleFileToolGenerationRuntime {
    let metadataClient: ToolMetadataClient
    let languageModel: LanguageModel
    let cleanup: ContentViewSourceCleanup
    let repairLoop: ContentViewBuildRepairLoop
    let bundleClient: ToolAppBundleClient
    
    // 主入口：Create Mode
    func generateTool(from userPrompt: String) async -> GenerationResult {
        // Stage 1: Metadata
        let metadata = await metadataClient.generateMetadata(from: userPrompt)
        
        // Stage 2: Optional Refinement
        let refinedPrompt = await promptRefinement(userPrompt)
        
        // Stage 3: Content Generation
        let rawCode = await generateContentView(
            prompt: refinedPrompt,
            metadata: metadata
        )
        
        // Stage 4: Source Cleanup
        let cleanedCode = cleanup.clean(rawCode)
        
        // Stage 5: Compilation + Repair Loop
        let (finalCode, success) = await repairLoop.buildAndRepair(
            code: cleanedCode,
            packageName: metadata.displayName
        )
        
        guard success else {
            return .failure(.repairFailed)
        }
        
        // Stage 6: App Bundle Packaging
        let appBundle = await bundleClient.buildBundle(
            code: finalCode,
            metadata: metadata
        )
        
        return .success(appBundle)
    }
    
    // 核心：代码生成
    private func generateContentView(
        prompt: String,
        metadata: ToolMetadata
    ) async -> String {
        
        let systemPrompt = ToolGenerationPrompts.singleFileCodingInstructions
        
        let userPrompt = """
        Build a macOS utility app: \(metadata.displayName)
        
        \(prompt)
        """
        
        return await languageModel.chat(
            system: systemPrompt,
            user: userPrompt
        )
    }
}

完整系统提示词（40+ 约束）

这是 Ironsmith 架构的核心。以下提示词基于 AGENTS.md 中的约束列表反推，包含所有框架规则、代码组织、负约束和输出格式要求。

系统提示：单文件 SwiftUI 编码指令（完整版）
// ============================================================
// Ironsmith ToolGenerationPrompts.singleFileCodingInstructions
// 约 40 条约束，定义 LLM 的输出边界
// ============================================================

You are an expert SwiftUI developer for macOS 26.0+.
Your task is to write a single, complete SwiftUI view file.

=== FRAMEWORK RULES ===
1. Write exactly ONE file named ContentView.swift
2. Define exactly ONE struct: struct ContentView: View
3. Do NOT create @main, App, Preview, or #Preview blocks
4. Do NOT use ObservableObject, @Published, @StateObject, or @ObservedObject
5. Keep ALL state with @State properties inside ContentView
6. Do NOT create additional files or modules
7. Do NOT append @main to any struct
8. Platform: macOS 26.0+ only (not iOS, not iPadOS)
9. Prefer Apple platform frameworks (Foundation, SwiftUI, AppKit) over 3rd-party

=== CODE ORGANIZATION ===
10. Use these section MARKs when possible:
    // MARK: - State
    // MARK: - Body
    // MARK: - Actions
    // MARK: - Helpers
11. Keep the Body compact; extract helper views and methods
12. All state must be declared in ContentView, not top-level

=== FUNCTIONAL CONSTRAINTS (NEGATIVE) ===
13. Do NOT add analytics, telemetry, or tracking
14. Do NOT add backend, server, or API calls unless explicitly requested
15. Do NOT add iCloud, CloudKit, or sync features
16. Do NOT add auth, login, or user accounts
17. Do NOT add onboarding, tutorials, or help screens
18. Do NOT add settings/preferences unless explicitly requested
19. Do NOT make anything overly complex — code must fit in a single file
20. Do NOT add network requests unless the user explicitly asked for data fetching

=== STATE MANAGEMENT ===
21. Use @State for local view state
22. Use @State var isPresented for sheet/modal state
23. Use @FocusState for focus management
24. Use AppStorage ONLY for small user preferences (not primary data)

=== UI/UX PATTERNS ===
25. Prefer native macOS controls: List, Form, Toolbar, Menu
26. Use .navigationTitle() for window titles
27. Use .frame(minWidth:idealWidth:maxWidth:) for responsive layout
28. Use proper SwiftUI layout: HStack, VStack, ZStack, Grid
29. Support dark mode by default (system background colors)
30. Use SF Symbols for icons: Image(systemName: "...")

=== OUTPUT FORMAT ===
31. Return ONLY Swift source code
32. No markdown fences (no ```swift)
33. No explanation text before or after code
34. No thinking blocks or reasoning text
35. No comments explaining obvious code (self-documenting code preferred)
36. No TODO, FIXME, or placeholder comments
37. No imports beyond what's needed (remove unused imports)

=== COMPLEXITY BOUNDARY ===
38. If the user's request is too complex, simplify it to fit in a single file
39. Prefer focused utilities over full applications
40. One screen = one ContentView; do not create multi-screen navigation

约束分析

约束类型 数量 目的
框架规则 9 确保代码在 Ironsmith 的固定脚手架内运行
代码组织 3 统一代码风格，便于解析和修复
负约束（不要做什么） 8 防止 LLM 过度工程化，保持单文件
状态管理 4 统一使用 @State，避免复杂模式
UI/UX 6 确保 macOS 原生体验
输出格式 7 确保输出可被机器解析，减少 cleanup 工作量
复杂度边界 3 防止 LLM 生成超出单文件能力的内容

为什么 "负约束" 比 "正约束" 更重要

LLM 的训练数据包含大量 "完整应用" 的代码，所以 LLM 有强烈的倾向去添加：

  • Preview 块（方便开发调试）
  • App 结构（完整的应用入口）
  • ObservableObject（MVVM 模式）
  • 网络请求（从后端获取数据）
  • Analytics（追踪用户行为）
  • Onboarding（首次使用引导）

但在 Ironsmith 的架构中，这些都是 不允许的。因为：

  • Preview 和 App 结构由平台固定脚手架提供
  • ObservableObject 在单文件模式下增加复杂度
  • 网络请求需要权限配置，超出单文件范围
  • Analytics 和 Onboarding 不是核心功能

🔑 核心洞察： Ironsmith 的提示词设计不是 "教 LLM 怎么写 SwiftUI"，而是 "定义 LLM 的输出边界"。约 60% 的约束是 "不要做什么"，而不是 "要做什么"。

如果我们要为 LongMarch 设计类似的系统提示，应该：

LongMarch 单文件 HTML 编码指令（参考设计）
=== FRAMEWORK RULES ===
1. Write only the content inside <body> tags
2. Do NOT include <html>, <head>, <body> tags — the platform provides them
3. Do NOT include <script src="..."> for external libraries — use inline JS only
4. Do NOT include CSS link tags — the platform provides Tailwind CSS

=== CSS RULES ===
5. Use the platform's CSS variables: --primary, --surface, --accent, --text
6. Prefer inline <style> or Tailwind classes over external CSS
7. Dark theme is default; use color: var(--text) for text

=== JS RULES ===
8. Use vanilla JavaScript only (no frameworks)
9. All code in a single <script> block at the end of body
10. Do NOT use eval() or document.write()

=== NEGATIVE CONSTRAINTS ===
11. Do NOT add backend, server, or API calls
12. Do NOT add analytics, tracking, or cookies
13. Do NOT add auth, login, or user accounts
14. Do NOT add external links or redirects
15. Do NOT make overly complex — keep it focused

=== OUTPUT FORMAT ===
16. Return ONLY the HTML content (what goes inside <body>)
17. No markdown fences
18. No explanation text
19. Include both HTML structure and inline <style> + <script>
